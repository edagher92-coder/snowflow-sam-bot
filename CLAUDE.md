# CLAUDE.md — snowflow-sam-bot

Guidance for Claude Code (and other AI assistants) working in this repository.

## What this repo actually is (verified from code, not assumed)

This is a **Flask webhook server** (`app.py`) that answers inbound customer
messages for **Snow Flow Sydney** across three channels behind a single
`/webhook` endpoint: **Facebook Messenger, Instagram DM, and WhatsApp Cloud
API**. It is a distinct surface from the "Terminal bot" (`sam/sam_bot.py`)
documented in `edagher92-coder/Claude-code-`'s root `CLAUDE.md` §3 — that one
is a CLI (`claude-opus-4-8`-driven, logs unknowns to
`sam/open-questions.md`). This repo has no CLI entry point, no
`open-questions.md`, and no Claude-Opus dependency; it's a small,
self-contained Flask app meant to run as a long-lived HTTP service (e.g.
behind `gunicorn`, per `requirements.txt`), not invoked interactively.
**No Procfile/Dockerfile/render.yaml is checked in here**, so the actual
hosting target (Render, Fly, a VM, etc.) is not verifiable from this repo
alone — flag as unclear rather than assuming.

### How it works (`app.py`)
- Single Flask app, single `/webhook` route (`GET` for Meta's verification
  handshake using `VERIFY_TOKEN`; `POST` for inbound events from Messenger/IG
  and WhatsApp Cloud API — both land on the same endpoint, branched on
  `data.get("object")`).
- **Reply brain = `sam_replies.json`**, loaded once at import time from
  `REPLIES_FILE` env var (default: the file next to `app.py`). It supplies:
  - `rules` (22 keyword → canned-reply mappings),
  - `postbacks` (button/quick-reply payload → rule id or greeting),
  - `greeting_first_message` / `away_message` / `fallback` strings,
  - `business_hours` (AEST-offset gate controlling greeting vs away message),
  - `facts` + `llm_voice` blocks used only as grounding text for the optional
    LLM layer (never edited by the bot itself).
- **Routing order** (`decide()` in `app.py`): postback payload → keyword rule
  match in `sam_replies.json` → optional Anthropic LLM call (only if
  `ANTHROPIC_API_KEY` is set; system prompt in `build_system_prompt()` forces
  it to answer *only* from the `facts` block and never invent prices/specs/
  stock/policy) → first-message greeting/away-message → generic fallback.
- **Loop-proof by construction**: echoes (`is_echo`), delivery/read receipts,
  and attachment-only messages with no text are explicitly ignored, so the
  bot can never reply to its own message.
- `GET /` is a health-check endpoint (reports which channels are configured
  and how many rules are loaded) — not a UI.
- Env vars it reads: `PAGE_TOKEN`, `VERIFY_TOKEN`, `APP_SECRET` (optional
  `X-Hub-Signature-256` verification), `WHATSAPP_TOKEN`, `WHATSAPP_PHONE_ID`,
  `REPLIES_FILE`, `ANTHROPIC_API_KEY`, `ANTHROPIC_MODEL` (default
  `claude-3-5-haiku-latest`), `PORT` (default 8080).

### `sam_replies.json`
The single source of truth for what Sam says. Treat it like the reply-engine
pricing sources in the main monorepo: **prices/facts in `sam_replies.json`
are data, not something to paraphrase or invent** — if you need to change a
price, warranty term, or hours, edit this file directly (there is no build
step / generator for it in this repo, unlike `web/knowledge.json` in
`Claude-code-`). Keep `facts` accurate; it's fed verbatim into the LLM system
prompt as the only allowed grounding.

## `skill_router/` — vendored, not owned here

`skill_router/` in this repo is a **vendored copy** of the auto-registering
skill router. Per `edagher92-coder/Claude-code-`'s root `CLAUDE.md` §6, the
source of truth is `Claude-code-`'s `src/skill_router/` — **fix bugs there
first, then re-copy here.** Don't patch `skill_router/` in this repo directly
except as a stopgap; upstream the fix.

- `tests/test_skill_router.py` + `tests/conftest.py` exercise the vendored
  package (`conftest.py` just puts the repo root on `sys.path` so it imports
  without an install step).
- `.claude/skills/` here only carries the generic efficiency/router skills
  (`skill-router`, `auto-escalate`, `balanced`, `deep`, `quick`) — see
  `.claude/skills/INDEX.md` (auto-generated by
  `python -m skill_router --index`, don't hand-edit).

## Run / test

```
pip install -r requirements.txt   # flask>=3.0, gunicorn>=21.2
python app.py                     # dev server on :8080 (needs PAGE_TOKEN etc. in env)
pytest                            # runs tests/test_skill_router.py
```

There is no other test suite in this repo — only the vendored
`skill_router` tests exist; `app.py` itself has no automated tests checked
in here (flagging this rather than assuming coverage exists).
